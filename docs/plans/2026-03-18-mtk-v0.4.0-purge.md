# mtk v0.4.0: Archival Focus Purge

**Date:** 2026-03-18
**Status:** Design

## Motivation

mtk exists as part of the longecho personal archive ecosystem. Its purpose hierarchy:

1. **Archival**: faithful, durable preservation of email
2. **Search**: finding things in the archive
3. **Analysis**: understanding patterns (delegated to LLMs via MCP)
4. **MCP**: LLM access to the archive via pure SQL

With clerk handling day-to-day email (IMAP/SMTP bridge for Claude Code), mtk accumulated features that belong to clerk's domain or that are better handled by LLMs at query time. This purge removes those features and tightens the tool around its archival mission.

## Removals

### 1. IMAP push and bidirectional sync

**What:** `push.py` (tag writeback to IMAP), `ImapPendingPush` model, push queueing in tag commands, `mtk imap push` CLI command, bidirectional sync logic.

**Why:** mtk is an archive, not a sync tool. IMAP pull stays as an ingestion mechanism (it's the only standards-based way to extract email from Outlook 365 and similar providers). But writing tags back to the IMAP server is clerk's domain.

**Keep:** `auth.py`, `connection.py`, `pull.py`, `gmail.py`, `mapping.py`, `ImapSyncState`, `mtk imap` CLI commands (minus push).

**Affected call sites:**
- Delete `src/mtk/imap/push.py`
- Remove `queue_tag_change` calls from `tag_add` and `tag_remove` in `main.py` (lines ~1049-1054, ~1102-1107)
- Simplify `ImapSync` in `imap/__init__.py`: remove `push_only()`, remove push call from `sync()`, remove `ImapPendingPush` query from `status()`
- Remove `mtk imap push` CLI command from `imap_cli.py`

### 2. notmuch integration

**What:** Entire `src/mtk/integrations/` directory, `notmuch_app` CLI commands, `notmuch` optional dependency.

**Why:** clerk replaced the workflow that used notmuch. No longer part of the email toolchain.

**Cleanup:** Remove `notmuch` from `pyproject.toml` keywords, optional dependencies, `all` extra, and mypy overrides. Remove `MtkConfig.notmuch_config` field from `config.py`.

### 3. Day-to-day email commands

**What:** `inbox`, `show`, `thread`, `reply` CLI commands.

**Why:** These are email-client features. clerk handles reading/replying to email. mtk's CLI access points are `search` (find archived email), `import`/`export` (move data in/out), `tag` (classify), and `mcp` (LLM access).

### 4. Interactive shell

**What:** `shell.py`, `mtk shell` command.

**Why:** MCP server + CLI commands are sufficient. The REPL adds maintenance burden without clear archival value.

### 5. Privacy module

**What:** `privacy.py`, `PrivacyRule` model, `PrivacyConfig` (in `config.py`), `privacy_app` CLI commands, `privacy.yaml` config, `MtkConfig.default_export_allowed` field.

**Why:** Unused. Can be added back if needed.

**Dependency:** `export/base.py` imports `_email_to_dict` from `privacy.py` (used in `_apply_privacy()` for both filtered and unfiltered paths). Before deleting `privacy.py`, extract `_email_to_dict()` into `export/base.py`. Also remove `PrivacyFilter` parameter from all exporter constructors (including arkiv's TYPE_CHECKING import).

### 6. Person/PersonEmail/RelationshipAnalyzer

**What:** `Person` model, `PersonEmail` model, `email_recipients` association table, entire `src/mtk/people/` directory, `people_app` CLI commands.

**Why:** Identity resolution (merging email addresses into unified Person records) is fragile state that breaks on reimport. It's an analysis judgment better made by an LLM at query time via MCP SQL. The RelationshipAnalyzer methods are trivial SQL queries an LLM writes on demand.

**Migration:** Email currently has `sender_id` FK to Person. Replace with direct storage:
- `Email.from_addr` and `Email.from_name` already exist (no change needed for sender)
- Add `Email.to_addrs` (Text, comma-separated), `Email.cc_addrs` (Text, comma-separated), `Email.bcc_addrs` (Text, comma-separated) to replace `email_recipients` table
- Remove `Email.sender_id` FK

**Import pipeline updates required:**
- `_run_import_with_importer()` in `main.py` (lines ~688-700) must map `parsed.to_addrs`, `parsed.cc_addrs`, `parsed.bcc_addrs` to the new Email columns (join with commas)
- `pull.py`'s `_process_message()` (lines ~204-206) only parses `From` today; must also parse `To`, `Cc`, `Bcc` headers and store them
- `ParsedEmail` dataclass already has the right fields (no parser changes needed)

### 7. Annotation, Collection, CustomField models

**What:** `Annotation`, `Collection`, `CustomField` models, `collection_emails` association table.

**Why:** Lightly used or unused. Replaced by a JSON metadata column on Email for flexible, schema-free metadata that's queryable via `json_extract()`.

**Migration:** Add `Email.metadata` column (Text, JSON). This handles annotations, custom fields, and any future per-email metadata without schema changes.

### 8. Stats command

**What:** `mtk stats` CLI command.

**Why:** An LLM generates better, more flexible stats via `run_sql`. A fixed stats command is redundant with the MCP server.

### 9. Maildir importer

**What:** `maildir.py` importer, `mtk import maildir` CLI command.

**Why:** Maildir is easily converted to mbox. Three import formats (mbox, eml, gmail) plus IMAP pull cover all practical ingestion paths.

### 10. Tag source field cleanup

**What:** `Tag.source` currently defaults to `"notmuch"`.

**Why:** notmuch is being removed. Change default to `"mtk"`.

### 11. Email model cleanup

**What:** Remove `Email.export_allowed` (privacy module removed).

**Keep:** `Email.imap_uid`, `Email.imap_account`, `Email.imap_folder`. These are required by `pull.py` for UIDVALIDITY reset handling (`_clear_folder_state()` queries by account/folder to null out stale UIDs) and for deduplication during incremental pulls.

### 12. Search engine: fix `to:` operator

**What:** `SearchQuery.to_addr` is parsed but never used in query filters (marked TODO in `_like_search` and absent from `_fts5_search`).

**Fix:** With `Email.to_addrs` now a column, implement `to:` filtering as `Email.to_addrs.ilike(...)` in both search paths.

### 13. MCP server metadata cleanup

**What:** `TABLE_DESCRIPTIONS` and `QUERY_TIPS` in `server.py` reference removed tables (`persons`, `person_emails`, `email_recipients`, `annotations`, `collections`, `collection_emails`, `privacy_rules`, `custom_fields`, `imap_pending_push`).

**Fix:** Update to reflect post-purge schema. Add descriptions for new columns (`to_addrs`, `cc_addrs`, `bcc_addrs`, `metadata`).

### 14. Test cleanup

**What:** Test fixtures and test files reference removed models and commands.

**Delete:** `tests/test_people.py` (entire file).

**Update:**
- `tests/conftest.py`: remove `Person`, `PersonEmail`, `Annotation`, `Collection` from imports and `populated_db` fixture; remove `PersonFactory` and `mock_notmuch` fixtures; remove `sender_id` from email fixture data
- `tests/test_cli.py`: remove tests for `people`, `inbox`, `show`, `thread`, `reply`, `stats`, `shell` commands
- `tests/test_database.py`: remove tests for removed models

### 15. pyproject.toml cleanup

**What:** Remove `notmuch` keyword, `notmuch` optional dependency, update `all` extra, update project description (remove "relationship mapping"), clean mypy overrides for `notmuch2`.

## Data Model After Purge

```
emails
+-- id, message_id (unique), thread_id (FK)
+-- from_addr, from_name, to_addrs, cc_addrs, bcc_addrs
+-- subject, date, in_reply_to, references
+-- body_text, body_html, body_preview
+-- file_path
+-- imap_uid, imap_account, imap_folder
+-- metadata (JSON)
+-- created_at, updated_at

threads
+-- id, thread_id (unique), subject
+-- email_count, first_date, last_date
+-- created_at, updated_at

tags
+-- id, name (unique), source (default "mtk")

email_tags (association)
+-- email_id, tag_id

attachments
+-- id, email_id (FK), filename, content_type, size, content_id

imap_sync_state
+-- id, account_name, folder, uid_validity, last_uid, highest_modseq
+-- message_count, last_sync

emails_fts (FTS5 virtual table)
+-- subject, body
```

7 tables (6 regular + 1 FTS virtual), down from 16.

## CLI After Purge

```
mtk
+-- search QUERY
+-- init [path]
+-- mcp
+-- tag {add, remove, list, batch}
+-- rebuild {index, threads}
+-- import {mbox, eml, gmail, imap}

```

7 command entry points, down from 20+.

## What Stays (Unchanged)

- **FTS5 search engine**: core archival retrieval
- **MCP server**: `run_sql` + `get_schema` (pure SQL, the primary LLM interface)
- **All 5 export formats**: json, mbox, markdown, html SFA, arkiv JSONL
- **Import formats**: mbox, eml, gmail (Takeout), IMAP pull
- **Tagging system**: tags table + email_tags (classification is archival)
- **Threading**: deterministic from email headers
- **Attachments**: structural metadata about email content

## Migration Notes

This is a breaking schema change (v0.3.0 -> v0.4.0). No migration path; users reinitialize the database and reimport. This is acceptable because:
1. mtk is pre-release (no external users)
2. The archive can be rebuilt from source email files
3. The source data (mbox/eml/gmail exports) is the canonical store; mtk is a derived index
